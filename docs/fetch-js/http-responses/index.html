HTTP & Responses

Reading ⏱ 2 min read

Every time your browser asks a server for data, it follows the same pattern: your code sends an HTTP request, and the server sends back an HTTP response. Understanding this cycle is essential for working with fetch.

1. The Request-Response Cycle#

  1. Your JavaScript calls fetch(url) with the address of the API endpoint.
  2. The browser sends an HTTP request to the server at that address.
  3. The server processes the request and sends back an HTTP response.
  4. The response contains a status code and, usually, a body containing JSON data.
  5. Your code reads the status code to check whether the request succeeded, then converts the body to a JavaScript object.

2. HTTP Status Codes#

Every response includes a status code — a three-digit number that tells you whether the request succeeded or failed. Codes are grouped by their first digit:

HTTP status code ranges
RangeCategoryMeaning
1xxInformationalRequest received, processing continues
2xxSuccessRequest was received, understood, and accepted
3xxRedirectionFurther action is needed to complete the request
4xxClient ErrorThe request was invalid or cannot be fulfilled
5xxServer ErrorThe server failed to process a valid request

3. Common Status Codes#

Frequently encountered HTTP status codes
CodeNameWhat it means
200OKEverything worked — data is in the response body
201CreatedA new resource was successfully created
400Bad RequestThe request was malformed or missing required data
401UnauthorizedAuthentication is required or has failed
403ForbiddenYou are authenticated but don't have permission
404Not FoundThe requested resource does not exist
500Internal Server ErrorSomething went wrong on the server

4. Checking the Status in Code#

The Response object returned by fetch exposes two useful properties for checking the outcome:

  • response.status — the numeric status code (e.g. 200, 404)
  • response.ok — a boolean that is true when the status is in the 200–299 range
Important

fetch() only throws an error for network failures (e.g. no internet). A 404 or 500 response does NOT throw — you must check response.ok yourself.
